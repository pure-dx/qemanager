DX Quality Engineering team’s mission

Mission statement

The team’s mission could be summarized as:


Help the Digital Experience BU understand, improve and maintain high levels of mean effectiveness and efficiency of quality* assurance of its products.


* Quality defined as: The level of SW malfunction in expected standard use cases that has reached production environment

Team’s expected means to achieve the mission

The team is expected to achieve its mission via the following means, among others:

  • By defining quality metrics, KPIs, SLIs, scores that are meaningful and can be measured

  • e.g. component / requirement / scenario / feature set / … test coverage by: unit tests / integration tests / components tests / contract tests / E2E tests / …

  • By building means to measure the defined quality metrics (tooling integrated with CI/CD/monitoring, measurement & data collection frameworks, etc.)

  • e.g. code coverage classification (to distinguish different types of tests) and reporting framework, production deployment and rollback reporting, requirements framework usage reporting, 

  • By working with DX leadership to make the introduction and the maintenance of selected measurements mandatory (securing its place in DX teams roadmaps and routines)

  • e.g. enhancing reporting of CC to distinguish unit vs component tests, etc.

  • By working with DX leadership to introduce mandatory quality targets (securing its place in DX teams roadmaps and routines)

  • e.g. min code coverage, existence of at least some E2E / environment tests for all feature sets, etc.

  • By providing insight to DX teams and leadership about their organizations’ standing on quality metrics and quality targets

  • e.g. live dashboards with build / deployment / environment test run results, reports of current and historical quality levels on service / team level, regular reports of organization level to leadership, notifications about quality degradation, etc.

  • By providing tooling / frameworks / guidelines / best practices / consultancy / training / … to DX teams to help them implement the expected quality assurance means

  • e.g. E2E testing framework, environment test data preparation framework, scheduled environment testing framework, defining the deployment lifecycle, including the role of each application environment, requirement tracking and test coverage framework, demo data framework, etc. - all of which require documentation, training, consultancy to other teams that adopt them

  • When appropriate, by helping DX teams directly to adopt new tools / frameworks / … provided by DX QE team

  • e.g. set-up of some new DX QE-provided framework in a team’s repository, incl. some reference first usages

  • By creating and maintaining a set of top-level E2E automated tests that cover major DX-owned feature sets

Team’s success metrics

Assuming the team pursues the above mentioned means to achieve its mission,
the team’s impact can be measured using metrics centered around the delivery of the “means of improvement”, which may include, but are not limited to:

  • Monthly % adoption of each DX QE team-provided product

    • e.g. % of DX teams / feature sets / areas / services / repositories / … that use a DX QE team-provided dashboard / tool / framework / …

  • Monthly number of usages of each DX QE team-provided product

    • e.g. unique visits / invocations / code references / … of DX QE team-provided dashboard / tool / framework / …

  • Monthly number of DX QE team-defined and measured DX quality metrics, KPIs, SLIs, or scores where the measurement is no longer than 3 months old

  • Monthly % of DX quality metrics, KPIs, SLIs, or scores that have available DX QE team-provided technical means to measure them automatically

  • Monthly % adoption of each DX QE team-provided mean of measurement of DX quality metrics, KPIs, SLIs, or scores

    • e.g. % of relevant repositories that use a DX QE team-provided CI plugin that measures Unit Test coverage

  • Monthly number of DX QE team-provided top-level E2E automated tests of DX-owned feature sets

  • Monthly % coverage of DX-owned feature sets by DX QE team-provided top-level E2E automated tests

DX quality metrics

The insight provided by the team to DX leadership and teams about DX BU’s quality of products may include, but is not limited to, the following metrics:

  • Monthly number of functional bugs (and of what criticality) found in DX-owned feature sets in the production environment

  • Monthly number of rollbacks of DX-owned feature sets in the production environment

  • Monthly mean duration between the time when a functional bug of a DX-owned feature in the production environment is introduced and when it is identified

  • Monthly number of regressions (and at what level: same feature vs same feature set vs Pure1 ecosystem) in DX-owned feature sets in the production environment

  • Monthly % of feature sets that have defined requirements in a structured form

  • Monthly mean DX production component / requirement / scenario / … test coverage by: unit tests / integration tests / components tests / contract tests / E2E tests / … (definitions by Martin Fowler)

  • e.g. % of LoC covered / % of requirements covered / % of eligible repositories containing such tests / % of scenarios / …

Important notes

  1. DX teams consist only of generic SW Engineers - we do not have dedicated quality engineers in our organization. We expect all SW Engineers to own their team’s deliveries end-to-end, including quality assurance. DX QE team is a central team that supports the assurance of quality in our organization; however, they do not own it - instead, each DX product team owns the quality of services / feature sets they own. DX QE team only owns the tooling and reporting they build and provide.
